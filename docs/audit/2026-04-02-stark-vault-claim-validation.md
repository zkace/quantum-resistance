# STARK vault — code-backed claim validation matrix (2026-04-02)

This document is a **code-backed claim validation matrix for the STARK vault stack**. It maps **claims** appearing in the ZK-ACE guide, acquisition memo, scope/dataflow audit notes, and the in-repo paper PDF to **what the current STARK vault working-tree baseline actually enforces**. Verdicts are conservative: “supported” means the property is directly visible in Solidity/Rust/tests or clearly follows from standard STARK soundness assumptions; weaker claims are downgraded.

**Normative code paths:** `crates/zk-ace-stark`, `contracts/src/StarkVerifier.sol`, `contracts/src/StarkZkAceAccount.sol`, `vault-app/src/main.ts`, plus `docs/audit/2026-04-02-stark-vault-dataflow.md` for binding semantics.

**Verdict definitions:** `supported` = directly evidenced in current code/tests; `partially supported` = code points in that direction but depends on unstated assumptions, limited fixtures, or incomplete enforcement; `unsupported by current code/tests` = contradicted or not borne out by the current implementation; `requires external verification` = cannot be decided from this repository alone.

### Replay / security narrative — explicit mismatch summary

- **Collateral** (`ZK-ACE-Guide.md` §“Replay prevention”) implies **`zkNonce` monotonicity alone** prevents proof reuse.
- **Code** (`StarkZkAceAccount.sol`) verifies the STARK, then increments `zkNonce`, but **never** checks that the prover’s nonce field matches the current storage counter, nor recomputes `rp_com` in the EVM.
- **In-repo Solidity comments overstate enforcement:** NatSpec Step 9 describes verifying `rp_com` and even says it “changes with each transaction,” but the current EVM path does not compare `pubInputs[8]`, `pubInputs[9]`, `pubInputs[10]`, and `pubInputs[11]` against prior state or against a recomputed hash.
- **Audit dataflow** (`2026-04-02-stark-vault-dataflow.md`) documents **two nonce domains** (EntryPoint vs `zkNonce`) and flags **signature blob reuse** with fresh EntryPoint nonces and identical `callData` as a **review item**, not a closed invariant.
- **Concrete replay shape to review:** same `signature` bytes + fresh EntryPoint nonce + unchanged `callData`.

Treat replay protection as **compositionally** relying on STARK soundness + public-input binding + **ERC-4337 nonce rules**, not on a single Solidity equality check.
