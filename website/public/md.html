Part 0: The Cryptography Landscape

Everything you need to hold your own in a conversation about cryptography, quantum threats, and where ZK-ACE sits in the bigger picture.

How modern cryptography works

All of cryptography rests on one-way functions — things that are easy to compute forward but hard to reverse. There are two families:

Symmetric cryptography (one key for both sides):

  • AES (encryption), SHA-256, Keccak256 (hashing), HMAC (authentication)
  • Security assumption: the functions behave like random oracles — no shortcut to invert them
  • Quantum impact: Grover's algorithm gives a quadratic speedup (square root), so 256-bit security becomes 128-bit. Still safe — just double your key sizes.

Asymmetric cryptography (public key + private key):

  • RSA, Diffie-Hellman, ECDSA (what Ethereum uses), EdDSA
  • Security assumption: specific math problems are hard — factoring (RSA), discrete logarithm (DH), elliptic curve discrete logarithm (ECDSA)
  • Quantum impact: Shor's algorithm solves ALL of these in polynomial time. Complete break. Not "weakened" — broken.

This is the core asymmetry: quantum computers destroy asymmetric crypto but only mildly dent symmetric crypto.

The three hard problems that underpin today's internet

Problem Used by Classical hardness Quantum hardness
Integer factoring RSA Sub-exponential (hard) Polynomial — broken by Shor
Discrete logarithm (finite fields) Diffie-Hellman Sub-exponential (hard) Polynomial — broken by Shor
Elliptic curve discrete log (ECDLP) ECDSA, EdDSA, BN254 pairings Exponential (very hard) Polynomial — broken by Shor

Every blockchain, every TLS connection, every SSH session, every encrypted message relies on at least one of these. All three fall to the same algorithm.

Shor's algorithm — what it actually does

You studied math, so here's the actual mechanism:

1. Shor reduces factoring/DLP to period-finding in a group

2. Period-finding is done via the Quantum Fourier Transform (QFT), which is efficient on a quantum computer

3. For an n-bit key, Shor needs roughly 2n+3 logical qubits and O(n^3) gates

4. For ECDSA-256 (Ethereum): ~500 logical qubits, but each logical qubit requires ~1,000-10,000 physical qubits for error correction

5. So you need roughly 500K-5M physical qubits to break Ethereum keys

Current state of quantum hardware (2026):

  • IBM: 1,386 qubits (Flamingo, 2025), roadmap to 100K+ by 2033
  • Google: Willow chip (105 qubits, 2024), demonstrated "below threshold" error correction
  • Others: IonQ, Quantinuum, PsiQuantum all progressing

Credible timeline for breaking ECDSA: 2029-2035. Not certain, but not speculative either. This is a "when" problem, not an "if" problem.

Grover's algorithm — the symmetric threat

Grover gives a quadratic speedup for unstructured search. If a hash function has 2^n possible outputs, Grover finds a preimage in 2^(n/2) time.

  • SHA-256: 256-bit → 128-bit post-quantum. Still safe.
  • AES-128: 128-bit → 64-bit post-quantum. Not safe. Use AES-256.
  • Keccak256: 256-bit → 128-bit post-quantum. Still safe.

This is why ZK-ACE uses keccak256 (256-bit hash) — even with Grover, you get 128-bit security, which is the standard target.

Why NIST PQ signatures don't solve the blockchain problem

The signature sizes tell the story:

  • ECDSA signature: 64 bytes
  • ML-DSA (Dilithium) signature: 2,420 bytes (38x larger)
  • SLH-DSA (SPHINCS+) signature: 7,856 bytes (123x larger)

On Ethereum L1, calldata costs 16 gas per non-zero byte. On L2 rollups, calldata is the dominant cost because it gets posted to L1. A 38x increase in signature data means a 38x increase in the authorization portion of transaction costs.

This is the gap ZK-ACE fills: instead of a large post-quantum signature, you submit a compact STARK proof (~44 KB) that achieves quantum resistance without the calldata bloat of lattice signatures. And unlike hash-based signatures (SPHINCS+, Lamport), STARKs are stateless — the same key works forever.

The "hardness assumptions" hierarchy

Different PQ approaches rely on different math:

Hash-based (STARKs, Lamport, SPHINCS+) Hash functions are one-way Highest — studied for 50+ years, no structural weaknesses
Lattice-based (Dilithium, FALCON, Kyber) Shortest Vector Problem (SVP) in lattices High — studied for 25+ years, but newer than hash-based
Code-based (McEliece) Decoding random linear codes High — 40+ years, large keys
Isogeny-based (SIDH/SIKE) Supersingular isogeny walk Broken — SIKE was broken in 2022 by a classical attack

ZK-ACE sits in the hash-based category — the most conservative, most battle-tested foundation. The only assumption is that keccak256 and Rescue-Prime behave as random oracles.

The blockchain quantum threat landscape

What's at risk:

1. All EOA wallets — private keys recoverable from public keys via Shor's algorithm

2. All smart contract wallets using ECDSA — same issue

3. SNARK-based ZK rollups — proofs rely on BN254 pairings (elliptic curves)

4. Cross-chain bridges — typically secured by multisigs (ECDSA)

5. TLS/HTTPS — the connections to RPCs use key exchange that's quantum-vulnerable (being migrated to ML-KEM)

What's NOT at risk:

1. Hash-based commitments — Merkle trees, state roots, etc.

2. Symmetric encryption — AES-256 with 128-bit PQ security

3. Proof-of-work (for chains that use it) — mining uses SHA-256 (symmetric)

4. STARK-based proofs — hash-only, no elliptic curves

The "harvest now, decrypt later" attack:

Adversaries (nation-states) are recording encrypted traffic today, planning to decrypt it when quantum computers arrive. For blockchains, this is less relevant because transactions are public — but the ability to derive private keys from public keys (which are visible on-chain after the first transaction) means an attacker with a quantum computer could drain any wallet that has ever transacted.

Ethereum's response

The Ethereum Foundation formed a Post-Quantum team in January 2026. Their roadmap:

1. Short term: Account abstraction (ERC-4337) as a migration path — let wallets upgrade their signature schemes without changing the protocol

2. Medium term: leanVM/leanSig — new precompiles for efficient PQ signature verification in the EVM

3. Long term: Protocol-level migration away from ECDSA for validator signatures

ZK-ACE is essentially an implementation of option 1 — using ERC-4337 to replace ECDSA with quantum-safe STARK proofs, without waiting for protocol changes.

Part 1: Why This Exists

The problem in one sentence

Every Ethereum wallet uses a digital signature scheme called ECDSA, which relies on the hardness of a math problem (the discrete logarithm on elliptic curves) that quantum computers will be able to solve efficiently.

Why that matters

Ethereum wallets work like this today:

1. You have a private key (a random 256-bit number)

2. From it, you derive a public key (a point on an elliptic curve)

3. To authorize a transaction, you produce an ECDSA signature using your private key

4. Anyone can verify the signature using your public key

The security assumption: given the public key, you can't figure out the private key. This is the Elliptic Curve Discrete Logarithm Problem (ECDLP) — hard for classical computers, easy for quantum computers via Shor's algorithm.

Timeline estimates for a quantum computer capable of breaking ECDSA: 2027-2033. When that happens, every wallet on Ethereum, Bitcoin, Solana — everything using elliptic curves — becomes vulnerable.

What ZK-ACE does differently

Instead of signing transactions with a private key, you prove in zero knowledge that you know a secret, without ever revealing the secret or using any elliptic curve math.

The proof uses only hash functions (specifically keccak256, the same hash built into the Ethereum Virtual Machine). Hash functions are symmetric cryptography — Shor's algorithm doesn't help. Grover's algorithm (the best quantum attack on symmetric crypto) only halves the security: 256-bit becomes 128-bit, which is still unbreakable.

Part 2: The Core Idea

Identity, not keys

Traditional wallet: "I have the private key that matches this public key."

ZK-ACE vault: "I know a secret that is consistent with a commitment stored on-chain, and here's a mathematical proof of that — without revealing the secret."

The components:

Concept What it is Analogy
REV (Root Entropy Value) The prover witness derived from your 24-word mnemonic via PBKDF2, then reduced into the Goldilocks field for the STARK circuit. The combination to your safe
IDcom (Identity Commitment) Hash(REV, salt, chainId). Stored on-chain. The lock on the safe — everyone can see the lock, but not the combination
STARK Proof A mathematical proof that you know the REV matching the on-chain IDcom. Proving you know the combination without opening the safe in front of anyone

The hash function: Rescue-Prime

To compute IDcom from REV, we use Rescue-Prime (Rp64_256), a hash function designed for arithmetic circuits. It operates over the Goldilocks field and outputs 4 field elements (256 bits total).


IDcom = RescuePrime(REV, salt, chainId)
     → 4 Goldilocks elements
     → packed as 32 bytes (bytes32 in Solidity)

Why Rescue-Prime instead of keccak256 for commitments? Because Rescue-Prime is "algebraically friendly" — it's efficient to prove inside a STARK circuit. Keccak256 is used for the Merkle trees and Fiat-Shamir challenges (the outer proof structure), but the identity commitment itself uses Rescue-Prime because that's what gets proven.

Part 3: The Goldilocks Field

All arithmetic in ZK-ACE happens in the Goldilocks field:


p = 2^64 - 2^32 + 1 = 18,446,744,069,414,584,321

This is a prime number slightly below 2^64. It's called "Goldilocks" because it sits in a sweet spot:

  • Small enough that two elements can be multiplied without overflow in a 128-bit register (64 × 64 = 128)
  • Large enough for meaningful security when using 4 elements together (4 × 64 = 256 bits)
  • Has nice roots of unity — 2^32 divides (p-1), so there are 2^32-th roots of unity, which makes FFTs (Fast Fourier Transforms) efficient

The field also has a quadratic extension: we adjoin an element t satisfying t^2 - t + 2 = 0, giving us pairs (a, b) representing a + bt. This doubles the security per element and is used throughout the STARK proof.

Part 4: STARKs — How the Proof Works

What is a STARK?

Scalable Transparent ARgument of Knowledge.

  • Scalable: Verification time is polylogarithmic in the computation size
  • Transparent: No trusted setup (unlike Groth16/SNARKs which need a toxic-waste ceremony)
  • Argument of Knowledge: Proves you know a witness (the REV) satisfying certain constraints

STARKs are built on three ideas from math:

1. Polynomials — The witness is encoded as polynomial evaluations

2. Reed-Solomon codes — Low-degree polynomials are extended to a larger domain to enable spot-checking

3. Merkle trees — Commitments to polynomial evaluations via hash-based data structures

The AIR (Algebraic Intermediate Representation)

The computation we want to prove is encoded as a table (the "execution trace") with constraints:


18 columns × 8 rows

Columns 0-3:    IDcom    (4 Goldilocks elements)
Columns 4-7:    target   (4 Goldilocks elements)
Columns 8-11:   rpCom    (replay-prevention commitment)
Column 12:      domain   (chain ID)
Columns 13-16:  txHash   (4 Goldilocks elements)
Column 17:      step     (counter: 0, 1, 2, ..., 7)

Transition constraints (what must be true between consecutive rows):

  • Columns 0-16: each value must stay the same (next == current)
  • Column 17: counter must increment by 1 (next == current + 1)

Boundary constraints (what must be true at specific rows):

  • At row 0: all 17 public input values must match the trace
  • At row 0: step counter must equal 0

This seems almost trivially simple — the trace is constant except for a counter. The real cryptographic work happens in how the trace is constructed (the prover must know the REV to compute the correct IDcom, target, and rpCom via Rescue-Prime hashing). The constraints guarantee the prover committed to a valid computation.

The proof pipeline

Here's what happens when you authorize a transaction:


1. PROVER (your browser, via WASM)
   ├── Inputs: REV (secret), salt, chainId, nonce, txHash
   ├── Compute: IDcom = Rescue(REV, salt, chainId)
   ├── Compute: derivedKey = Rescue(REV, algId, chainId, index)
   ├── Compute: target = Rescue(derivedKey)
   ├── Compute: rpCom = Rescue(IDcom, nonce)
   ├── Build execution trace (18 × 8 table)
   ├── Extend trace to LDE domain (18 × 64, blowup factor 8)
   ├── Commit to trace via Merkle tree (keccak256)
   ├── Evaluate constraints, commit via Merkle tree
   ├── DEEP composition (algebraic linking of evaluations)
   ├── FRI low-degree test
   ├── Proof-of-Work (20-bit grinding)
   └── Output: ~44 KB proof + 17 public inputs

2. VERIFIER (Solidity smart contract on Arbitrum)
   ├── Recompute txHash from calldata (never trust the prover)
   ├── Check IDcom matches on-chain storage
   ├── Check domain matches block.chainid
   ├── Rebuild Fiat-Shamir transcript (derive all challenges)
   ├── Verify OOD (Out-of-Domain) constraint evaluation
   ├── Verify Proof-of-Work
   ├── For each of 44 queries:
   │   ├── Verify Merkle proofs (trace + constraints)
   │   ├── Compute DEEP composition
   │   └── Check against remainder polynomial
   ├── If all checks pass → valid proof
   └── Increment zkNonce, return SUCCESS

Key concepts in the proof

Fiat-Shamir transform: In an interactive proof, the verifier sends random challenges. Fiat-Shamir makes this non-interactive by deriving challenges from a hash of everything seen so far (the "transcript"). The verifier recomputes the same hash, getting the same challenges. This is why the proof is non-interactive — the prover and verifier agree on randomness without communicating.

Low-Degree Extension (LDE): The 8-row trace is interpolated as polynomials and evaluated at 64 points (8× blowup). This redundancy lets the verifier spot-check: if the polynomial is low-degree (consistent with a valid trace), random evaluations will match. If the prover cheated, they'd need to forge a high-degree polynomial that passes random checks — exponentially unlikely.

DEEP composition: Instead of checking each column separately, the verifier combines all columns into one "DEEP" polynomial using random coefficients. This reduces 18 separate checks to 1, without losing soundness (by the Schwartz-Zippel lemma — a non-zero polynomial of degree d over a field of size p has at most d/p chance of evaluating to zero at a random point).

FRI (Fast Reed-Solomon IOP of Proximity): Proves that the committed polynomial is actually low-degree. In our case with 0 FRI folding layers, this reduces to checking against a remainder polynomial.

Proof-of-Work (grinding): 20-bit PoW on the Fiat-Shamir transcript. The prover must find a nonce such that keccak256(seed || nonce) has 20 leading zero bits. This adds ~20 bits of security and prevents certain algebraic attacks.

Part 5: Security Properties

Where does 128-bit post-quantum security come from?

Multiple layers contribute:

Component Classical security Post-quantum security
Rescue-Prime commitments (4 elements) 256 bits 128 bits (Grover halves)
STARK soundness (44 queries, blowup 8) 132 bits 132 bits (not affected by quantum)
Proof-of-Work 20 bits 20 bits
Keccak256 (Merkle trees) 256 bits 128 bits (Grover)
Goldilocks field (per element) 64 bits 32 bits (but never used alone)

The bottleneck is 128-bit post-quantum from the hash functions — well above the standard 128-bit security target.

What exactly is proven?

The STARK proof demonstrates:

"I know values (REV, salt) such that:
- Rescue(REV, salt, chainId) = the IDcom stored on-chain
- Rescue(Rescue(REV, algId, chainId, index)) = target
- Rescue(IDcom, nonce) = rpCom
- And these values are consistent with the transaction I'm authorizing"

Without revealing REV or salt.

Replay prevention

Each transaction uses a monotonically increasing zkNonce. The replay-prevention commitment rpCom = Rescue(IDcom, nonce) is part of the STARK statement, and the contract increments zkNonce after each verifier-accepted proof. In the current implementation, replay resistance is the combination of that proof statement, callData binding, and ERC-4337 nonce behavior rather than a direct on-chain recomputation of rpCom.

Cross-chain replay prevention

The domain field is set to block.chainid at deployment. A proof generated for Arbitrum (chainId 42161) won't validate on Ethereum mainnet (chainId 1) because the domain check fails.

Why the core authorization path avoids ECDSA

The entire authorization chain is:


BIP-39 mnemonic (24 words, 256-bit entropy)
  → PBKDF2-HMAC-SHA512 (symmetric KDF, quantum-safe)
    → REV (identity secret)
      → Rescue-Prime hash (algebraic hash, quantum-safe)
        → STARK proof (hash-based proof, quantum-safe)
          → Keccak256 verification (EVM-native hash, quantum-safe)

Zero elliptic curves. Zero pairings. Nothing for Shor's algorithm to attack.

Part 6: The Smart Contract Architecture

ERC-4337 Account Abstraction

ZK-ACE uses ERC-4337, the Ethereum standard for smart contract wallets. Instead of EOAs (Externally Owned Accounts) that sign with ECDSA, ERC-4337 lets smart contracts define their own validation logic.

The flow:


User (browser)
  → Builds a UserOperation (like a transaction, but for smart wallets)
  → Generates STARK proof
  → Puts proof in the UserOp's "signature" field
  → Sends to Bundler (Pimlico, Alchemy, etc.)
    → Bundler submits to EntryPoint contract
      → EntryPoint calls vault's _validateSignature()
        → Vault decodes proof + 17 public inputs
        → Vault recomputes txHash from calldata
        → Vault checks IDcom, domain
        → Vault calls StarkVerifier.verifyProof()
        → If valid → execute the transaction

Identity rotation

If you suspect your mnemonic is compromised:

1. Generate a new mnemonic → new REV → new IDcom

2. Call proposeIdentityRotation(newIdCom) (requires valid proof)

3. Wait 48 hours (timelock — gives you time to cancel if someone else proposed)

4. Call confirmIdentityRotation() (requires valid proof)

The zkNonce is NOT reset during rotation — this prevents replay attacks across identity boundaries.

Emergency pause

setPaused(true) freezes all vault operations. Critically, setPaused bypasses the pause check itself — so you can always unpause by submitting a valid proof, even while paused.

Part 10: Questions You'll Get Asked (and Answers)

Q: "Why not just use NIST post-quantum signatures like Dilithium or FALCON?"

A: Those signatures are 2-5 KB each. On L2 rollups where calldata costs dominate, that's a 30-40x increase in authorization data cost. STARKs are more efficient because a single compact proof replaces the signature entirely, and the verification uses the EVM's native keccak256 opcode (30 gas).

Q: "Why Goldilocks field instead of BN254?"

A: BN254 (used by Groth16/SNARKs) relies on elliptic curve pairings — quantum-vulnerable. Goldilocks is a prime field with no elliptic curve assumptions. It also has efficient 64-bit arithmetic and nice FFT-friendly properties.

Q: "Why 8 rows? That seems tiny."

A: The trace length is independent of security. Our AIR is simple (constant columns + counter) because the hard work (Rescue-Prime hashing) happens during trace construction, not in the constraints themselves. The 8-row trace is sufficient to encode all the public inputs we need to verify. Security comes from the number of FRI queries (44) and the blowup factor (8), not the trace length.

Q: "What if quantum computers break keccak256?"

A: Keccak256 is a symmetric primitive. The best known quantum attack (Grover) reduces 256-bit security to 128-bit. Breaking 128-bit symmetric crypto would require roughly 2^128 quantum operations — this is not feasible even with theoretical future quantum computers. The cryptographic community consensus is that 128-bit symmetric security is safe post-quantum.

Q: "Why STARKs over SNARKs?"

A: SNARKs (like Groth16) require a trusted setup ceremony that produces "toxic waste" — if not properly destroyed, it can be used to forge proofs. STARKs are transparent: no trusted setup, no toxic waste, no trust in any third party. Additionally, Groth16 uses BN254 pairings (elliptic curves, quantum-vulnerable). STARKs use only hash functions.

Q: "How is this different from just running a hash-based signature on Ethereum?"

A: Hash-based signatures (like Lamport, SPHINCS+) are stateful — each key can only be used once or a limited number of times. ZK-ACE is stateless: the same REV can authorize unlimited transactions. The proof proves knowledge of the REV without consuming it. Additionally, on-chain verification of raw Lamport signatures costs ~24 KB of calldata per transaction. Our STARK proof is ~44 KB.

Q: "Is the roughly $0.20 cost competitive?"

A: A normal ECDSA transaction on Arbitrum costs roughly ~$0.01-0.02 at typical fee levels. So ZK-ACE is materially more expensive per transaction. The tradeoff is quantum resistance. For high-value assets (DAO treasuries, large holdings), paying roughly $0.20 for proof-backed authorization can still be a reasonable insurance premium. As L2 fees change, this number moves with them.

Q: "What happens if I lose my mnemonic?"

A: Same as any wallet — your funds are lost. The identity rotation feature lets you migrate to a new mnemonic while still in control, but once you lose access entirely, there is no recovery. This is the fundamental tradeoff of self-custody.

Q: "Could someone front-run the proof generation?"

A: No. The proof is bound to a specific transaction (txHash is a public input recomputed from calldata on-chain). A proof for one transaction cannot authorize a different transaction. The domain binding (chainId) prevents cross-chain replay, and same-chain replay resistance comes from the combined STARK statement, zkNonce progression, and ERC-4337 nonce behavior.

The three-layer PQ roadmap

The EF thinks about PQ in three layers, each with different solutions:

Consensus Layer (validator signatures):

  • Current: BLS signatures (96 bytes, aggregate to 96 bytes for 10,000 validators)
  • Problem: BLS uses elliptic curve pairings → broken by Shor
  • Solution: leanXMSS (hash-based signatures, ~3,000 bytes each)
  • Aggregation: leanMultisig (a minimal zkVM that proves "I verified N valid signatures" in a single STARK proof)
  • Result: 10,000 signatures → 125 KB (250x compression)
  • Status: 4 devnets shipped, 10 client teams building, targeting L/M fork (~2029)

Execution Layer (user wallets):

  • Current: ECDSA (secp256k1, 64-byte signatures)
  • Problem: ECDSA → broken by Shor
  • Solution: Account abstraction (ERC-4337) to let wallets choose their own signature verification
  • Future: PQ signature precompiles (native EVM opcodes for ML-DSA/Falcon/SPHINCS+ verification)
  • Status: Account abstraction is live. Precompiles are milestone J* on the roadmap (~2029).
  • ZK-ACE sits here — we're the only deployed implementation of the AA-based PQ migration path.

Data Layer (blob proofs):

  • Current: KZG commitments (elliptic curve based)
  • Problem: KZG → broken by Shor
  • Solution: Replace with PQ-safe commitment scheme
  • Status: Research phase, targeting M* fork

leanSig vs ZK-ACE — different problems, different layers

This is the most important distinction to understand:

leanSig (EF) ZK-ACE (us)
Layer Consensus (validators) Execution (user wallets)
What it replaces BLS validator signatures ECDSA wallet signatures
Signature scheme XMSS (hash-based, stateful) STARK proofs (hash-based, stateless)
Aggregation leanMultisig (zkVM) Not needed (one proof per tx)
Who uses it Validator node operators (~1M) End users (everyone)
Deployed? Devnet Mainnet
Compete? No — different layers entirely

The key talking point: "leanSig secures Ethereum's consensus. ZK-ACE secures Ethereum's users. Both are needed. They don't overlap."

The leanVM zkVM

leanVM is a purpose-built minimal zkVM for aggregating hash-based signatures:

  • 4 opcodes (extremely minimal — Cairo has hundreds)
  • Built on Plonky3 (newer STARK framework by Polygon) and WHIR (efficient polynomial commitment)
  • Uses multilinear STARKs and logup lookups
  • Current recursion time: 2.7 seconds, targeting 0.27 seconds
  • Purpose: aggregate thousands of leanXMSS signatures into one proof per slot

This is NOT a general-purpose zkVM. It does one thing: aggregate signatures. It's not a competitor to ZK-ACE — it operates at a completely different level (consensus infrastructure vs. user wallet).

  • On-chain verification of ML-DSA/Falcon will cost ~100-200K gas (cheap) instead of ~500M gas (current Solidity)
  • Direct PQ signatures become viable for user wallets
  • Our Solidity STARK verifier (5.6M gas) becomes comparatively expensive

This is a real constraint on long-term value. But:

Part 14: Bitcoin's Quantum Problem

Useful context for broader conversations about PQ crypto.

Why Bitcoin is harder than Ethereum

Bitcoin has no smart contracts, no account abstraction, no way to add custom signature verification without a protocol-level soft fork. Every solution requires consensus changes that take 5-10 years (based on SegWit and Taproot precedent).

What's vulnerable

  • ~1.7M BTC in P2PK addresses (public key fully exposed, including Satoshi's ~1.1M BTC)
  • ~5.2M BTC in reused addresses
  • ~35% of total Bitcoin supply is at risk
  • Google's March 2026 research: real-time attacks could hijack in-flight transactions in ~9 minutes

BIP-360 (the leading Bitcoin proposal)

  • New output type: P2MR (Pay-to-Merkle-Root), SegWit v2, addresses start with bc1z
  • Supports ML-DSA, SPHINCS+, Falcon signatures
  • BTQ Technologies has a working testnet (50+ miners, 200K+ blocks)
  • Estimated 5-10 years for full activation

The signature size problem for Bitcoin

Scheme Signature Txs per block
Schnorr (current) 64 bytes ~10,000+
ML-DSA-44 ~3,227 bytes ~1,000
SPHINCS+-128s ~7,856 bytes ~490

PQ signatures would crush Bitcoin's throughput. The only solution: STARK-based signature aggregation (StarkWare's S2morrow project) — batch all PQ signatures in a block into one STARK proof. This requires OP_CAT activation (its own soft fork battle).

Why this matters for your conversations

When people ask "what about Bitcoin?" your answer is: "Bitcoin needs protocol-level changes that will take 5-10 years. Ethereum's account abstraction lets us solve it today at the wallet level, without waiting for protocol upgrades. That's the advantage of the EVM."
